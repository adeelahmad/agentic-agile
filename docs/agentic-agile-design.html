01The two boundaries

Every decision in this design hangs off two lines.

The human boundary. Humans interact only during planning. Execution is autonomous. This resolves the platform's "no mid-run user input" limit cleanly — steering was never needed mid-run, because planning is the checkpoint and execution is hands-off.

The phase boundary. Planning is interactive and human-gated; execution is autonomous and hook-enforced. The hooks are the substitute for the human attention deliberately absent during execution — a skill is advisory, a hook fires regardless of the model, so the determinism the design needs only exists once the skill ships bundled with its gates. That bundling is the whole reason the artifact is a plugin.

Mapped to ICE — Intent · Context · Expectations

A useful lens (from intent-driven development) over artifacts we already have — a lens, not a renaming. The human owns Intent and Expectations at planning; the harness owns Context and feeds it progressively.

CraftOwnerWhere it lives
Intent — what's wanted, constraints, failure + success scenarios, connectionsthe humanintake.md + stories.md (the five parts)
Expectations — the done-ness boundary, in the user's termsthe same humanstories.md DoD + demo (of record); validate.md/plan.md derived
Context — the how, fed progressivelythe harnessstandards.md → plan-ready.md → init.md + the symbol resolver

Presence → hooks. The human owns Expectations at planning, then steps out of execution; the gates enforce that human-authored done-ness on every attempt. So "presence in the loop" becomes mechanical and continuous — not a single human blessing of an oversized diff at the end. That substitution is sound only because Expectations are machine-checkable (validate.md is literal PASS/FAIL). Completeness of Intent stays the irreducible human judgment — which is why the five-part Intent gate exists.

02Lifecycle

Planning produces validated artifacts; execution walks waves through RED → scaffold → GREEN → GATE.

Planning runs the five steps and hands off only when every Stage-2 artifact validates. Execution is autonomous: per wave, RED writes failing tests, a scaffold pass fires after RED is verified to create canonical stubs, GREEN fills them, and a final GATE certifies the sprint. Failures the supervisor cannot resolve surface at the next planning session — the only path back to a human.

Placing the scaffold pass after RED keeps the playbook's "RED fails because no production code exists" rule untouched, while still giving GREEN canonical, single-source stubs to fill — so two GREEN workers cannot produce two divergent implementations.

Review corrections

The flow below incorporates fixes from a structural review of an earlier draft: the reviewer renamed to structural-reviewer, remediation paths added to the structural and final gates, cause-split escalation re-entering at the planner, idempotent scaffolding, standards.md bound to the execution gates, and a no-progress (livelock) guard. See DESIGN.md §12.

03The eight personas

Each activity is its own folder under pipeline/<phase>/<NN-activity>/ holding the same four descriptive files.

The folder is the documentation: persona.md (who the orchestrator dispatches), init.template.md (the contract handed in), artifacts.md (what it reads / writes and what validates each), and gate.md (the orchestrator's acceptance check plus failure verbs).

PhasePersonaIdentityMandate (one line)

planningstandardsthe lawkeeperassemble the exact rule set this sprint must obey
planningplannerthe architectrun the five planning steps; own every planning artifact
executionred-workerthe falsifierwrite the tests that must fail; one per task
executionscaffolderthe carpenterafter RED, stub every referenced symbol exactly once
executiongreen-workerthe minimalistfill the stub bodies with the least code that passes
executionstructural-reviewerthe auditorcatch orphan modules, parallel impls, duplicate helpers

Adopted

intake now carries the five-part Intent — what's wanted, constraints, failure scenarios, success scenarios, and connections — and the planner gate rejects a story missing failure-scenarios or connections. This makes the most-skippable judgment (front-end intent completeness) hard to skip; everything downstream of it is already hook-enforced.

Note

The structural-reviewer is the one role with no prior explicit spec — it is derived from the roster plus the post-mortem of a real project where two implementations of one engine shipped undetected. It is the check that would have caught that.

04Artifact ownership

The supervisor owns every planning artifact; sub-agents never edit them.

Sub-agents never edit stories.md, tasks.md, validate.md, plan.md, plan-ready.md, or sprintN/plan.md. What each writer may touch:

WriterMay write
supervisorevery planning artifact · plan-ready.md · execution.log
red-workertest files + mod common shims only
scaffolderproduction stubs (panic + TODO) only · deletes shims · appends ledger
green-workerproduction bodies (fills TODOs) + removes shim leftovers
structural-reviewer / final-gatetheir output.md report only (read-only over code)
Expectations of record

The user-facing definition of done — stories.md's Definition of Done and Sprint demo — is owned by the same human who owns the Intent, and never drifts to the agents. validate.md and plan.md are the derived, implementation-level expressions of it. The moment "done" drifts away from the person who wanted the outcome, the agent starts deciding done for them — which is exactly what the ownership rule and the planning boundary prevent.

05The revision loop

init.md and output.md fill the playbook's "reject and re-spawn" gap.

The playbook rejects a bad attempt and re-spawns, but defines no feedback payload — a blind re-spawn repeats the mistake. init.md is the per-attempt contract (supervisor / scaffolder authored, append-only, turn-counted, read-only to the worker); it points at the relevant plan-ready.md section and, on re-spawn, carries a feedback entry derived from the prior output.md. output.md is the worker's own report.

Scope distinction so nothing duplicates: init.md is per-task-attempt (context + feedback), output.md is per-task-attempt (the report), plan-ready.md is the per-story static spec written once, and execution.log is the per-sprint one-line index.

06Supervisor judgment

On an unresolvable gate failure, the supervisor chooses — against a written rule.

The supervisor classifies each failure and logs the choice: halt the dependency chain when the failure poisons a shared foundation (e.g. two divergent engines for one abstraction), or continue independent work and collect the escalation when the failure is isolated. The test is simply: does the unresolved failure sit on a dependency that other pending work needs? Every choice is written to execution.log — auditable and replayable — and surfaces at the next planning session for the human to confirm or override.

07Validation model

md-db / KDL validates the markdown artifacts; grep-gates cover what it structurally cannot.

md-db validate <dir> --schema <x>.kdl enforces YAML frontmatter, field types, enums, patterns, required sections, and cross-document reference integrity. It cannot enumerate dynamically-named sections — the playbook's ## Tn — Title headings — so validation is two-layered: schemas for structure, bin/ grep-gates for the dynamic body shape.

SchemaScopeCoverage
agent-io.kdlper-task-attempt comms — init.md, output.mdfully validated (new artifacts, fixed sections)
planning-artifacts.kdlstories / tasks / validate / plan / plan-ready / sprint-planfrontmatter + fixed sections (see OPEN-7)
ledger.kdlexecution.logwrapping doc; line format enforced by bin/

08Symbol-resolution backend

One subsystem of a prior build is harvested to back the two gates md-db and grep cannot do well.

Two gates need source-code intelligence that operates on code, not markdown: gate-scaffold-verify ("is this symbol defined exactly once?") and gate-structural-integrity ("two definitions filling one role?"). grep is fragile here — it matches comments, misses overloads, and cannot see semantic duplication. So the code-intelligence layer of the earlier build is harvested; everything else is left out.

Integration is optional with a fallback so the plugin never hard-fails: the two gates call the symbol resolver if it is present, else fall back to grep with a logged "weaker check" warning. md-db (over the markdown artifacts) and the symbol resolver (over the source code) are complementary and independently optional; neither drags in the orchestration engine.

11Open questions

The forks that still block a final build. Two were resolved this round.

OPEN-1

Scaffolder = the supervisor itself, or a dispatched stub-only sub-agent (which needs its own "implemented a body → reject" guard)?

OPEN-2

Does skills/ become the canonical copy of the playbook (replace), or call out to the existing doc (wrap)? Biggest layout impact.

OPEN-3

Does the Stage-2-complete gate run as a hook, or as a supervisor self-check step? (Planning is interactive, so a hook is awkward.)

OPEN-4

init.md / output.md path scheme — a per-attempt dir like sN-NN-<slug>/<task>/attempt-K/, or flatter?

OPEN-5

output.md — one file appended per attempt, or a fresh file per attempt?

OPEN-6

Max revision attempts before the supervisor escalates to the next planning session — what default?

OPEN-7

Add YAML frontmatter to stories.md / plan.md so md-db validates them structurally, or keep them frontmatter-free and grep-gate only?

OPEN-8

Leaning resolved. Symbol-backend packaging — recommendation is to extract a minimal ctx-symbols helper (locate_symbol / resolve_symbol only) rather than ship the whole prior binary.
